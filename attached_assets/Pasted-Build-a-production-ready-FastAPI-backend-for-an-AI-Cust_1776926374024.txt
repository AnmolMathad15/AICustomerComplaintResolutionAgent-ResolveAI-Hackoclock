Build a production-ready FastAPI backend for an AI Customer Complaint Resolution Agent called ResolvAI.

TECH STACK:
- Python 3.11+
- FastAPI + Uvicorn
- HuggingFace Transformers + PyTorch
- scikit-learn
- Pydantic for validation
- CORS enabled for all origins (frontend on localhost:3000)

AI MODELS (HuggingFace):
1. Complaint Classifier: facebook/bart-large-mnli
   - Zero-shot classification
   - Labels: ["billing issue", "refund request", "technical problem", "delivery complaint", "account access", "product quality"]
   - Returns: complaint_type, severity (HIGH/MEDIUM/LOW based on score), confidence score

2. Sentiment Analyzer: cardiffnlp/twitter-roberta-base-sentiment-latest
   - Returns: sentiment label, frustration score (0-100)
   - Escalation trigger: negative sentiment + score > 80

3. Summarizer: facebook/bart-large-cnn
   - Summarizes customer history + current complaint
   - Used only during escalation
   - max_length: 100, min_length: 30

CUSTOMER DATABASE (customers.json):
Create 6 detailed customer profiles with:
- customer_id, name, email, phone, tier (Premium/Standard/Basic), join_date
- history array with 3-5 past interactions each containing:
  ticket_id, channel (chat/email/phone), date, issue, complaint_type, status (resolved/pending/escalated), resolution, agent_name

CONFIDENCE SCORING LOGIC:
confidence_score = (classification_score * 0.5) + (policy_match_score * 0.3) + (history_factor * 0.2)
- policy_match_score: 1.0 if policy exists, 0.5 if partial match, 0.0 if no policy
- history_factor: 0.8 if customer has similar resolved issue, 1.0 if first time, 0.3 if repeated unresolved
- If final confidence < 0.60: flag for escalation

ESCALATION LOGIC:
Escalate if ANY of these conditions are true:
1. Sentiment frustration score > 80
2. Confidence score < 0.60
3. Complaint type matches escalate_conditions in policy
4. Customer has 2+ unresolved tickets of same type
Escalation response includes: summary (from summarizer model), assigned_agent, ticket_id, estimated_response_time

STARTUP BEHAVIOR:
- On server start: pre-load ALL 3 HuggingFace models into memory
- Print "ResolvAI Backend Ready" when all models loaded
- Store all analyzed complaints in memory (list) for dashboard stats
- Generate unique ticket IDs: TKT-{timestamp}-{random4digits}

ERROR HANDLING:
- If model inference fails: return mock response with confidence 0.5
- If customer_id not found: return generic customer template
- All endpoints return proper HTTP status codes
- Global exception handler returns { "error": "message", "status": "failed" }
